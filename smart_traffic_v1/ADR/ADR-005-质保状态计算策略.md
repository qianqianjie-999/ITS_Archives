# ADR-005: 质保状态计算策略

## 状态
已接受

## 背景
系统需要准确计算和管理各类交通设备的质保状态。设备可能关联到不同的项目，且支持质保延期操作。需要统一的逻辑来判断设备是在保还是过保。

## 决策
采用"有效质保到期日期"为核心的统一计算策略。

### 核心原则

1. **延期优先**：优先使用延期后的质保到期日期
2. **项目关联**：设备必须关联到项目，项目提供基础质保信息
3. **实时计算**：质保状态在查询时实时计算，不存储中间状态

### 质保状态类型

| 状态 | 含义 | 判断条件 |
|------|------|----------|
| 在保 | 设备在质保期内 | 有效质保到期日期 >= 当前日期 |
| 过保 | 设备已过质保期 | 有效质保到期日期 < 当前日期 |
| 无项目 | 设备未关联项目 | 无有效质保到期日期 |

### 有效质保到期日期计算

```python
@property
def effective_warranty_expire_date(self):
    # 1. 优先使用延期质保到期时间
    if self.extended_warranty_expire_date:
        return self.extended_warranty_expire_date
    # 2. 否则使用项目质保到期时间
    if self.project and self.project.warranty_expire_date:
        return self.project.warranty_expire_date
    return None
```

### 质保状态判断

```python
@property
def warranty_status(self) -> str:
    expire_date = self.effective_warranty_expire_date
    if expire_date:
        if expire_date >= date.today():
            return '在保'
        else:
            return '过保'
    return '无项目'
```

### 设备类型与质保延期

| 设备类型 | 质保延期字段 | 支持延期 |
|---------|------------|---------|
| TrafficLight | extended_warranty_expire_date | ✅ |
| ElectronicPolice | extended_warranty_expire_date | ✅ |
| ParkingEnforcement | extended_warranty_expire_date | ✅ |
| Checkpoint | extended_warranty_expire_date | ✅ |
| BackendDevice | extended_warranty_expire_date | ✅ |

### 路口/点位混合状态计算

路口和点位可能包含多个设备，每个设备可能有不同的质保状态：

```python
@property
def warranty_status(self) -> dict:
    expire_dates = []
    for device in self.devices:
        expire_date = device.effective_warranty_expire_date
        if expire_date:
            expire_dates.append(expire_date)

    if not expire_dates:
        return {'warranty_status': '无项目', 'latest_expire_date': None}

    today = date.today()
    in_warranty = sum(1 for d in expire_dates if d >= today)
    expired = sum(1 for d in expire_dates if d < today)

    # 混合状态判断
    if expired == 0:
        status = '在保'
    elif in_warranty == 0:
        status = '过保'
    else:
        status = '混合'

    latest = max(expire_dates)
    return {
        'warranty_status': status,
        'latest_expire_date': latest.isoformat()
    }
```

### 混合状态类型

| 状态 | 含义 | 判断条件 |
|------|------|----------|
| 在保 | 所有设备都在质保期内 | expired == 0 |
| 过保 | 所有设备都已过质保期 | in_warranty == 0 |
| 混合 | 部分在保，部分过保 | 其他情况 |

### 质保延期流程

1. 用户发起延期申请
2. 系统创建新的项目记录（或使用现有项目）
3. 更新设备的 `extended_warranty_expire_date`
4. 创建质保延期记录（WarrantyExtension）
5. 后续查询自动使用延期后的日期

### WarrantyExtension 表设计

```python
class WarrantyExtension(db.Model):
    facility_type: Mapped[str]  # 'intersection', 'point', 'traffic_light' 等
    facility_id: Mapped[int]     # 设备ID
    project_id: Mapped[int]      # 项目ID
    extension_date: Mapped[date] # 延期日期
```

## 替代方案考虑

| 替代方案 | 优点 | 缺点 |
|---------|------|------|
| 存储计算后的状态 | 查询快速 | 需要同步更新，可能不一致 |
| 定时任务计算 | 可批量处理 | 实时性差 |
| 视图/Materialized View | 自动计算 | 增加数据库复杂度 |

## 后果

### 正面
- 质保状态实时准确
- 延期操作灵活，不影响原有项目数据
- 统一的计算逻辑，易于维护
- 便于追溯延期历史（WarrantyExtension表）

### 负面
- 每次查询都需要计算（可通过缓存优化）
- 设备表增加延期字段（轻微数据冗余）
- 混合状态可能让用户困惑（需要UI说明）
